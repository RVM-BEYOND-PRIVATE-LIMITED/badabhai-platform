# Migration Plan (Phase 1)

## Source of truth

- **Schema** is authored in Drizzle: `packages/db/src/schema.ts`.
- **Migrations** are generated by Drizzle Kit into `packages/db/migrations/`.

## Recommended flow (app development)

```bash
# 1. Edit packages/db/src/schema.ts
# 2. Generate a migration (no DB connection needed)
pnpm db:generate
# 3. Review the generated SQL in packages/db/migrations/*.sql
# 4. Apply it
pnpm db:migrate           # uses DATABASE_URL (local docker or Supabase)
```

`DATABASE_URL` may point at:
- local docker Postgres (`docker compose up -d`), or
- a Supabase project's connection string (Project Settings → Database).

## If Supabase should own migration history

Some teams prefer Supabase CLI to track migrations (for `supabase db push` /
branch previews). In that case:

1. Generate SQL with Drizzle (`pnpm db:generate`).
2. Create a Supabase migration and paste the SQL:
   ```bash
   supabase migration new init_worker_profiling
   # copy packages/db/migrations/0000_*.sql into the new supabase/migrations file
   supabase db push
   ```
3. Pick **one** authority per environment — do not double-apply.

## Generating Supabase TypeScript types (optional)

After the schema exists in a linked project:

```bash
supabase gen types typescript --project-id <PROJECT_REF> > packages/db/src/supabase.types.ts
```

These are complementary to Drizzle's inferred types; Drizzle row types remain the
primary types used by services.

## Conventions

- Status columns are `text` (not pg enums) for migration simplicity. Add `CHECK`
  constraints in a follow-up migration if/when the value sets stabilize.
- Never write a destructive migration without an explicit, reviewed plan.
- Keep migrations forward-only in shared environments.
