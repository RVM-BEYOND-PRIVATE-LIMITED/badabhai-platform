# ADR 0001 — MVP Infrastructure & AI Decisions

- **Status:** Accepted
- **Date:** 2026-06-08
- **Phase:** 1 (Worker Profiling)

## Context

BadaBhai is an AI placement-team product for blue/grey-collar India, starting
with industrial manufacturing (CNC/VMC). Phase 1 must ship a privacy-safe worker
profiling + profile generation flow with a lean team, while keeping the door open
for the deterministic Reach Engine, employer/unlock, and payments in later phases.

## Decisions

1. **Supabase Postgres is the MVP database.** Managed Postgres with Auth/Storage
   we can grow into. Schema is authored in Drizzle (`packages/db`); migrations are
   generated by Drizzle Kit and can be applied via Drizzle or Supabase CLI.
2. **Google Cloud credits are for experiments/staging/AI tests only** — not the
   production system of record in Phase 1. Avoids premature lock-in.
3. **AI is API-first.** All LLM access is mediated by a LiteLLM adapter.
4. **No self-hosted LLM at launch.** Operationally cheaper and faster to iterate;
   revisit only if cost/latency/privacy demands it.
5. **A pseudonymization gateway is mandatory before every LLM call.** It runs in
   the FastAPI AI service and **fails closed**. Phone, full name, address,
   employer names, and ID-doc tokens never reach an LLM.
6. **Flutter for the worker app.** One codebase for Android-first reach.
7. **Worker profiling first.** Phase 1 builds identity → consent → chat profiling
   → extraction → resume. 
8. **Employer posting, unlock, payments, payouts, boosts, ranking/matching are
   later.** The `@badabhai/reach-engine` package is a deliberate placeholder.
9. **Legal/compliance flows are placeholders for now.** DPDP consent is captured
   structurally; production legal copy/flows come later (`docs/legal-later`).

## Consequences

- **Positive:** fast to build, privacy-safe by construction, low ops burden,
  clean seams (events, packages) for future phases.
- **Negative / risks:** Supabase RLS is not finalized in Phase 1 (backend uses
  the service role; see `infra/supabase/rls-plan.md`); pseudonymization is
  heuristic in Phase 1 (over-masking is the safe direction); mock OTP/STT/LLM
  paths must be replaced before any real launch.
- **Reversibility:** API-first AI and the LiteLLM adapter make switching model
  providers cheap. Drizzle schema is portable off Supabase if needed.
