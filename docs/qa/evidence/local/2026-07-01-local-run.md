## Root cause — stale local DB (environment, not code)

The local `badabhai` DB has **drifted** from the migration journal and is ~1 week behind `main`:

| Signal | Value |
| --- | --- |
| Tables in `public` | 30 |
| Drizzle `__drizzle_migrations` entries applied | 26 |
| Latest applied migration timestamp | 2026-06-23 |
| `pnpm db:migrate` | ❌ fails `42P07` (relation already exists — journal/DB out of sync) |
| Missing tables (`main` expects) | `worker_flags`, `worker_devices`, `worker_credentials` (added by post-2026-06-23 migrations — ADR-0026 device/PIN + flags) |

The 14 failures are entirely explained by those 3 missing tables: the RLS-spine test
enumerates all tables and asserts REVOKE on each, so the missing tables error out. Every
functional flow that doesn't touch those tables passed. **No application-code defect is
implicated.**

## Not auto-fixed (flagged for owner)

Reconciling the local DB was **not** performed — it is potentially destructive (the journal
disagrees with the DB, so a clean fix is a fresh-DB re-migrate or a mark-applied + apply-the-3-
missing-tables, either of which the owner should authorize; no prod/shared data). The cloud
Supabase DB was reconciled to `main` on 2026-06-27; the **local** DB was not and remains stale.

Recommended local reconcile (owner's call): migrate a **fresh** DB from `0000` (clean journal,
no drift) and point the API at it, or reconcile the existing DB's `__drizzle_migrations` before
`db:migrate`. After that, re-run `RUN_E2E=1 pnpm --filter @badabhai/e2e test` to clear the 14
RLS-spine failures.
