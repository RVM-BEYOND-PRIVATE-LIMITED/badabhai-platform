| R11 | **Worker real name (TD21) is now served over unauthenticated endpoints** — `GET /resume/:id` returns the name inside the resume body, and `PUT /workers/:id/name` lets any caller set a worker's name; neither has caller-identity authz (the platform-wide gap, R1/TD4) | Medium | Open | Bounded for now: ids are unguessable UUIDv4; `generated_resumes`/`workers` are RLS+REVOKE-locked at the DB (TD20) so the Data API can't read them; no in-repo client consumes these. | R12 | **`voice_notes.transcript_text` is plaintext at rest** — raw worker free-text (can contain phone/name/employer) is stored unencrypted, unlike `phone_e164`/`full_name` (now AES-256-GCM) | Medium | Open | Contained: `voice_notes` is RLS+REVOKE-locked (migration 0009, TD20) so not Data-API readable; no app endpoint returns the transcript; it never enters events/ai_jobs/logs and reaches the LLM only via the pseudonymization gate. **Fix before real voice profiling:** encrypt `transcript_text` with `PiiCryptoService` (same as phone/name). Found in the [PII-at-rest/RLS review](../security/phase-1-pii-at-rest-rls-review.md) (2026-06-12). | Security + Backend |
| R13 | **Resume PDF render serves a worker's real name via signed-URL download** (`GET /resume/:id/download`, TD5/[ADR-0007](../decisions/0007-resume-render-node-boundary.md)) with no caller-identity authz (the R1/TD4 gap) | High | Open | Bounded: UUIDv4 object keys; `RESUMES_BUCKET` is private + backend service-role only; the `generated_resumes` row is RLS+REVOKE-locked (TD20); the signed-URL TTL is ≤15 min (`RESUME_SIGNED_URL_TTL_SECONDS`=900), minted server-side per request, never logged or put in an event; the name is HTML-escaped into the PDF and **renders in Node, never ai-service** (ADR-0007). **Launch gates:** (1) close caller authz (R1/TD4); (2) confirm `worker-resumes` is created **PRIVATE** out-of-band — migration 0009 RLS does NOT cover Storage object ACLs (same caveat as R10); (3) DPDP erasure-on-revoke must sweep the `resumes/{workerId}/` prefix (single sweep, like R10). While TD4 is open, `RESUME_GLOBAL_DAILY_CAP` is the only spend backstop (the per-worker cap is bypassable by rotating `worker_id`). | Security + Backend |
